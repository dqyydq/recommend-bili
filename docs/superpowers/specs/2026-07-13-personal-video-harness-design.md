# Favorite Agent Personal Video Harness Design

## 1. 产品定位

Favorite Agent 是一个本地优先、可自行部署的开源个人视频知识助手。B 站收藏是第一个数据源，但系统边界按可扩展 Harness 设计，未来可以通过适配器接入网页书签、YouTube、RSS 等内容源。

产品承诺不是“一次性整理收藏夹”，而是持续完成以下闭环：

1. 同步用户主动收藏的内容。
2. 理解当前问题、长期兴趣和正在推进的目标。
3. 从个人内容库检索证据并提出可执行行动。
4. 从用户的点击、跳过、完成和纠正中更新记忆。
5. 周期性提供回顾、继续学习和安全清理建议。

核心价值表达：

> 它记得你收藏过什么、正在学什么、经常放弃什么，并持续把收藏转化为检索结果、学习进度和可确认的行动。

## 2. 产品原则

- **本地数据优先**：PostgreSQL 是业务数据源，Chroma 只存检索向量。用户自行配置模型密钥，默认不上传遥测数据。
- **记忆透明**：Agent 记住的事实、偏好和经验必须可查看、纠正、删除和暂停更新。
- **证据优先**：检索回答必须引用真实收藏条目。证据不足时明确表达不确定性。
- **自动分析、人工执行**：读取、分析、索引和生成草稿可以自动运行；删除、移动等外部修改必须经过用户确认。
- **渐进式上下文**：只注入当前任务需要的画像与记忆，避免把全部历史对话塞进模型上下文。
- **时间感知**：收藏和历史行为只证明用户“曾经感兴趣”，不能自动当作当前偏好。画像必须区分当前、降温、休眠和历史兴趣。
- **开放可替换**：模型提供商、Embedding、内容适配器和 Skill 都通过稳定接口替换，不把产品绑定到单一服务。

## 3. 长期使用循环

系统以四种可重复价值维持使用，而不是依赖功能数量：

- **每日价值**：结合时间预算和当前目标，推荐今天值得看的内容。
- **项目价值**：保存学习任务、对话上下文、完成状态和周期回顾。
- **记忆价值**：逐步理解主题偏好、内容难度、视频时长倾向和推荐禁区。
- **维护价值**：持续发现失效、重复、长期积压和结构问题，并产生待确认操作。

工作台每次打开应回答四个问题：

1. 自上次使用后发生了什么？
2. 当前最值得继续的目标是什么？
3. Agent 为什么给出这些建议？
4. 哪些操作正在等待用户确认？

## 4. Harness 架构

Harness 负责组织模型、上下文、记忆和工具，而不是让前端根据关键词直接决定调用哪个 API。

### 4.1 核心组件

#### Session

一次连续对话或任务运行。保存用户输入、Agent 回答、工具调用摘要、引用证据和运行状态。Session 是追加式记录，旧事件不被覆盖。

#### Context Builder

为每次模型调用组装一个有预算的上下文包：

- 当前输入与最近对话。
- 当前学习项目或待审核计划。
- 与当前任务相关的长期记忆。
- 检索得到的视频证据。
- 可用工具及权限边界。

Context Builder 必须限制每类信息数量，并记录实际注入了哪些记忆，便于解释和调试。

#### Memory Service

管理三种长期记忆：

- **语义记忆**：稳定事实和偏好，例如“偏好实战型教程”。
- **情景记忆**：用户行为与结果，例如“连续跳过三次 40 分钟以上的视频”。
- **程序记忆**：Agent 应遵循的个性化规则，例如“未知失效视频不要自动选中”。

每条记忆包含内容、类型、来源、置信度、有效时间范围、创建时间、最后确认时间和状态。低置信度推断不会作为确定事实展示。

#### Planner / Router

将任务路由到边界明确的 Skill：

- `retrieve_favorites`
- `analyze_topics`
- `manage_learning_project`
- `inspect_collection_health`
- `detect_expired_videos`
- `draft_folder_structure`
- `review_user_memory`

第一阶段采用单个 Planner 加确定性工具协议，不引入多个自由对话 Agent。后续可以增加独立 Evaluator，但工具权限仍由 Harness 控制。

#### Tool Registry

每个工具声明名称、用途、输入模式、输出模式、风险等级和是否需要确认。风险等级固定为：

- `read`：只读，可自动执行。
- `draft`：生成本地草稿，可自动执行。
- `mutate`：修改 B 站数据，必须确认。

#### Run Log

记录每次 Harness 运行的目标、路由、工具、记忆引用、证据、耗时、结果和错误。用户看到的是简化后的操作记录，开发者可查看结构化调试详情。

### 4.2 运行流程

1. 创建或恢复 Session。
2. Planner 判断任务意图和所需工具。
3. Context Builder 读取短期上下文和相关长期记忆。
4. 工具召回内容、分析主题或生成草稿。
5. Retrieval Reranker 根据当前问题、画像、项目和历史行为重新排序证据。
6. 模型生成带引用的回答或行动草稿。
7. Harness 运行输出检查，确认引用存在且危险动作未被直接执行。
8. 保存运行记录，并异步提取可能的新记忆。
9. 用户反馈后更新记忆置信度和推荐策略。

## 5. 长短期记忆

### 5.1 短期记忆

短期记忆属于 Session，包含：

- 最近的用户与 Agent 消息。
- 最近检索过和推荐过的视频 ID。
- 当前指代对象，例如“第二个视频”。
- 当前任务、过滤条件和未完成下一步。
- 当前学习项目 ID。

达到消息预算时生成结构化摘要，而不是永久保留全部消息进入上下文。原始记录仍可存档，但默认不注入模型。

### 5.2 长期记忆写入

长期记忆不从每句话无条件提取。以下事件触发候选记忆：

- 用户明确表达偏好或禁区。
- 用户纠正 Agent 的判断。
- 同一行为模式出现至少两次。
- 学习项目创建、完成、阻塞或复盘。
- 用户对推荐执行观看、跳过或完成反馈。

候选记忆先去重、合并并计算置信度。用户明确表达的规则可以直接激活；行为推断初始为低置信度，并在画像页标记为“Agent 推测”。没有发生行为不能单独成为负面偏好的证据，例如长时间没看篮球视频不能直接推断用户已经不喜欢篮球。

### 5.3 时间有效性与兴趣漂移

用户兴趣不是永久标签。系统将兴趣记忆区分为四种状态：

- `active`：近期明确表达或多次行为支持的当前兴趣。
- `cooling`：近期信号减少，但尚不能判断已经结束。
- `dormant`：用户明确表示不再关注，或长期出现可靠的反向信号。
- `historical`：作为人生经历保留，但默认不影响当前推荐。

不同记忆采用不同的衰减策略：

- 用户明确设定的安全规则和推荐禁区不自动衰减，直到用户修改。
- 学习项目目标随项目状态结束，不再作为全局兴趣长期加权。
- 从收藏和点击推断出的主题兴趣随时间缓慢降权。
- 用户明确表达“我现在不打篮球了”时，篮球兴趣立即转为 `dormant`，但不会删除历史收藏或抹掉过去画像。

系统在出现新旧信号冲突时不擅自覆盖：

- 明确陈述优先于行为推断。
- 新的明确陈述优先于旧的明确陈述，同时保留变更历史。
- 零散点击不能推翻稳定偏好。
- 连续出现的反向行为只降低置信度，达到阈值后提示用户确认“你的兴趣似乎发生了变化”。

检索默认只使用 `active` 和与当前问题直接相关的 `cooling` 兴趣。`dormant` 与 `historical` 兴趣只有在用户主动询问、搜索相关主题或使用“回看过去兴趣”场景时才参与推荐。用户本次明确提出的查询始终优先于画像，例如已经休眠的篮球兴趣不会阻止用户临时搜索篮球内容。

主题画像同时展示“当前兴趣”和“历史轨迹”。它可以表达“2023 年集中收藏篮球训练，2025 年后兴趣降温”，不能表达未经用户确认的原因，也不能推测受伤、疾病、家庭或经济状况等敏感事实。

### 5.4 记忆管理界面

“我的画像”展示：

- 当前兴趣主题及变化趋势。
- 降温、休眠和历史兴趣及其证据时间线。
- 内容形式和难度偏好。
- 正在推进的目标。
- Agent 学到的规则。
- 最近形成或改变的记忆。

每条记忆支持确认、纠正、标记为过时、恢复和删除。用户可以关闭行为学习或清空全部长期记忆，而不删除收藏数据。画像页应解释一条偏好为何处于当前状态，例如“最近 90 天没有新信号”或“你在 7 月 13 日明确表示不再关注”。

## 6. 智能检索与回答

### 6.1 召回与重排

检索分为两阶段：

1. Chroma 按问题 Embedding 召回候选，PostgreSQL 补充收藏夹、UP 主、时间和状态等业务字段。
2. Reranker 综合语义相关度、当前目标、时间有效的画像匹配、最近是否推荐、是否完成和内容新鲜度重新排序。

首版使用可解释的加权规则，保留每个信号分数；当数据积累充足后再考虑模型重排。画像信号必须附带时间权重，历史收藏数量不能压过近期明确偏好。用户明确提出“不要按我的偏好”时，只使用问题相关度。

### 6.2 上下文回答

回答格式包含：

- 直接结论。
- 推荐或判断依据。
- 引用收藏条目。
- 不确定性说明。
- 一个明确的下一步行动。

后端返回受限 Markdown，前端通过安全 Markdown 渲染器显示标题、列表、强调、链接和行内代码。禁止原始 HTML，并对链接协议做白名单过滤。

连续追问复用 Session，例如“换几个短一点的”“第二个适合初学者吗”“不要再推荐这个 UP 主”。

### 6.3 反馈闭环

检索结果支持“有用、忽略、已看、稍后”。反馈同时更新短期上下文和情景记忆，但不会直接修改原始收藏元数据。

## 7. 主题分析重构

现有主题分析的“EventSource 抓取全部收藏 -> KMeans -> LLM 命名”与本地同步架构重复。新流程只读取 PostgreSQL 快照：

1. 选择全部收藏或单个收藏夹。
2. 读取本地视频文本与现有 Embedding。
3. 对大数据集先进行抽样估计主题数量，再分批聚类，避免多次完整 KMeans 搜索。
4. 为每个主题生成名称、代表视频、主要 UP 主、活跃时间、近期变化和积压比例。
5. 将主题标记为增长、稳定、降温或历史兴趣，并生成主题之间的关系和可行动建议。
6. 保存分析版本，只有收藏快照明显变化时才重新计算。

结果界面采用“主题地图 + 主题列表”，不再把数千条视频一次性展开。点击主题后再加载代表内容和全部条目。

主题分析失败时保留上一次成功版本，并展示本次错误，不清空已有结果。

## 8. 收藏封面修复

封面链路包含三层保护：

1. 同步时规范化协议相对 URL、HTTP URL 和异常空值。
2. 前端使用固定比例容器、懒加载和加载失败占位，不显示浏览器破图图标。
3. 如果 B 站 CDN 防盗链仍导致失败，后端提供只允许已配置 B 站图片域名的代理端点，并设置 Referer、响应大小和超时限制，禁止成为通用代理。

代理缓存只保存短期图片响应，不把图片二进制写入 PostgreSQL。

## 9. 失效视频检测与一键清理

### 9.1 检测分类

检测 Agent 将视频分为：

- `confirmed_invalid`：B 站明确返回删除、失效或不存在，可默认选中。
- `review_required`：版权、地区、登录、风控或临时不可用，不默认选中。
- `unknown`：请求超时、限流或无法判断，不默认选中。
- `available`：正常可访问。

检测使用有限并发、指数退避和缓存。短时间内重复进入页面不重新请求全部视频。

### 9.2 清理流程

1. 用户启动扫描或查看最近扫描结果。
2. 界面默认选中 `confirmed_invalid`。
3. 用户查看标题、收藏夹、判定原因和最后检测时间。
4. 用户确认后生成 `mutate` 操作计划。
5. 服务端再次校验登录、CSRF 和目标清单后执行。
6. 每条成功、失败和跳过结果写入操作记录。

系统不承诺恢复 B 站已删除的收藏关系，因此确认对话必须清楚展示不可逆性。`review_required` 和 `unknown` 永远不会被“一键清理”自动选中。

## 10. 开源扩展模型

首版定义四类扩展接口：

- `ModelProvider`：DeepSeek、OpenAI-compatible 等聊天模型。
- `EmbeddingProvider`：远程 API 或本地模型。
- `ContentAdapter`：B 站及未来内容源的同步与状态检查。
- `AgentSkill`：工具描述、输入输出模式、权限和执行器。

扩展以 Python 协议和清晰示例为主，不在第一阶段建设插件市场或动态执行不受信任代码。社区 Skill 默认只读；申请 `mutate` 权限必须显式配置。

项目应提供：

- Docker Compose 一键启动 PostgreSQL 与应用依赖。
- `uv` 管理 Python 环境。
- `.env.example` 与模型提供商说明。
- 无真实账号也可体验的演示数据模式。
- 数据导出和记忆清空能力。
- 架构图、Skill 开发示例和安全边界文档。

## 11. 前端信息架构

保留现有一级导航，并调整内容职责：

- **工作台**：对话式任务入口、今日行动、当前项目、待确认操作和 Agent 最近学到的内容。
- **收藏库**：浏览、检索、主题地图和收藏健康度。
- **学习项目**：目标、任务、上下文对话和周期回顾。
- **操作记录**：同步、结构蓝图、失效清理和危险操作结果。

新增“我的画像”作为工作台中的上下文入口，而不是第五个同等级功能页。

结构蓝图历史条目必须可点击进入详情，查看建议节点、代表视频、审核状态和下一步操作。

## 12. 错误处理与安全

- 模型失败时返回可用的确定性检索结果，不能让整个页面空白。
- Chroma 不可用时回退到 PostgreSQL 文本检索，并明确提示语义能力降级。
- 所有后台任务保存状态、进度、超时、重试次数和可读错误。
- API Key 不进入日志、Prompt、操作记录或前端响应。
- 图片代理执行域名白名单、内容类型、大小和超时校验。
- 记忆内容视为不可信上下文，不得覆盖系统权限与工具规则。
- 模型不能自行提升工具权限，`mutate` 操作必须通过服务端确认令牌和用户会话校验。

### 12.1 画像与记忆边界情况

- **旧收藏不等于当前偏好**：收藏时间较早且没有近期信号时，只作为历史证据。
- **兴趣暂停不等于兴趣结束**：忙碌、季节变化或暂时没有新内容都可能造成沉默，系统不根据沉默直接下结论。
- **阶段目标不等于身份标签**：短期备考或工作任务结束后，对应主题停止全局加权。
- **误点和偶发探索**：一次点击、一次搜索或一次收藏只形成低权重情景记录，不直接改变画像。
- **矛盾偏好**：允许“喜欢篮球比赛，但不想看篮球训练”这类带条件偏好，不强行合并成单一主题标签。
- **多人或共享账号**：检测到长期明显冲突的内容模式时，只提示可能存在多个使用模式，不建立敏感身份推断。首版不自动拆分人物。
- **负面反馈作用域**：“不喜欢这个视频”默认只影响该条内容；只有用户明确表达主题、UP 主或形式禁区时才扩大范围。
- **历史回忆场景**：休眠兴趣可以用于年度回顾和用户主动回看，但不会出现在默认的今日推荐中。
- **敏感属性**：不得从收藏推断健康、宗教、政治、性取向、财务或家庭状况并写入画像，除非用户明确要求保存与当前任务必要的信息。
- **数据删除一致性**：删除收藏、项目、会话或记忆后，应清理对应向量与派生摘要，不能在后续回答中残留引用。
- **时间异常**：设备时区变化、缺失收藏时间和导入旧数据时，标记时间置信度，不伪造兴趣变化结论。
- **模型错误记忆**：只有结构化提取且能关联真实来源的候选记忆可以入库；无法定位来源的模型陈述不得写入长期记忆。

## 13. 验收标准

### 展示与数据

- 检索回答中的安全 Markdown 正确渲染。
- 收藏封面成功显示；失败时出现一致占位，不显示破图图标。
- 主题分析只读取本地快照，不触发全量 B 站抓取。
- 结构蓝图历史可以进入详情并继续审核。

### 记忆与检索

- 同一 Session 能理解“第二个”“换短一点”等追问。
- 新 Session 能根据已确认的长期偏好调整结果。
- 旧收藏形成的历史兴趣不会压过近期明确偏好。
- 用户明确表示兴趣结束后，该主题退出默认推荐但仍保留历史轨迹。
- 仅因长期没有点击，系统不会自动将兴趣判定为结束。
- 临时查询休眠主题时，系统仍能正常检索，不被画像错误过滤。
- 用户删除一条记忆后，后续上下文不再注入该记忆。
- 每个回答能追溯引用视频和使用的记忆摘要。
- 无模型或模型失败时仍返回基础检索结果。

### 安全清理

- 只有确定失效视频默认选中。
- 未知、限流和临时不可用视频不自动选中。
- 未确认前不调用 B 站删除接口。
- 批量执行结果逐项记录，部分失败可重试失败项。

### 开源可用性

- 新用户按照 README 能使用 `uv` 和 Docker Compose 启动项目。
- 用户可以配置至少一个 OpenAI-compatible 模型提供商。
- 演示模式无需 B 站账号和 API Key 即可浏览完整产品流程。
- 收藏数据、画像和记忆可以导出或清空。

## 14. 分阶段交付边界

### 第一阶段：修复基础体验

安全 Markdown、封面链路、结构蓝图详情、主题分析本地化，以及确定失效视频扫描与审核清理。

### 第二阶段：Memory 与 Session

会话记录、上下文摘要、长期记忆表、时间衰减、兴趣状态迁移、记忆提取、画像管理和行为反馈。

### 第三阶段：Harness 检索

Planner、Tool Registry、Context Builder、检索重排、运行日志和回答引用。

### 第四阶段：开源扩展与主动循环

模型与内容适配器接口、Skill 示例、演示模式、导出能力、每日建议和周期回顾任务。

第一阶段不引入自由运行的多 Agent、第三方 Skill 市场或无需确认的危险操作。多 Agent Evaluator 仅在单 Harness 运行数据与评估标准稳定后再考虑。
